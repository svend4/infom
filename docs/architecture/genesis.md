# Genesis — Порядок Творения AI-инфраструктуры

> «В начале было Слово» — и оно было весом в 70 миллиардов параметров.
> Но до Слова было Молчание — годы мышления, которые ждали своего инструмента.

Этот документ описывает эволюцию элементов AI-инфраструктуры:
от человеческого знания до полноценной когнитивной экосистемы.
Каждый слой появляется потому, что предыдущий обнаруживает свои пределы.

---

## День 0 — Человеческое Знание (Personal Archive)

**Что это:** До любой модели существует накопленный человеческий опыт —
идеи, диалоги, открытия, ошибки. Это сырьё, из которого строится всё.

**Проблема без AI:**
Знание хранится в голове, заметках, разрозненных файлах.
При переезде, болезни, утрате устройства — исчезает навсегда.
Третья попытка стоит дороже первой.

**Что нужно:** Инструмент, который принимает сырое человеческое знание
и делает его структурированным, запрашиваемым, бессмертным.

```
[Годы мышления] → [Диалоги / Заметки / Документы] → ??? → [Живая база знаний]
```

**Пример:** `data70` — 1105 ChatGPT-диалогов, 78 МБ, три попытки сохранить
(первая утеряна при госпитализации ~2010, вторая при эмиграции ~2022,
третья — этот репозиторий). Внутри: TetraDrone, WILOS, Legal AI, Zoo-роботы,
300+ диалогов по немецкому соцправу, 88-тематический каталог.

> **День 0 — это причина, по которой существуют все остальные дни.**

## День 1 — Языковая Модель (LLM)

**Что это:** Нейронная сеть, обученная предсказывать следующий токен.
Принимает текст → возвращает текст.

**Что умеет:**
- Отвечать на вопросы
- Переводить, резюмировать, генерировать
- Рассуждать в пределах контекстного окна

**Предел:** Контекстное окно конечно. Нет памяти между сессиями.
Нет доступа к внешнему миру. Нет возможности действовать.

```
Пользователь → [LLM] → Текст
```

**Примеры:** GPT-4, Claude, Gemini, Qwen, LLaMA, Mistral

---

## День 2 — Инструменты (Tools / Function Calling)

**Что это:** LLM получает возможность вызывать функции — Python-код,
API, браузер, файловую систему.

**Что добавляется:**
- Поиск в интернете
- Выполнение кода
- Чтение/запись файлов
- Вызов внешних API

**Предел:** Каждый вызов — одноразовый. Нет цикла. Нет планирования.
Модель не может сама решить «сделать несколько шагов».

```
Пользователь → [LLM + Tools] → Действие → Результат → Ответ
```

**Стандарт:** OpenAI Function Calling, Anthropic Tool Use, MCP

---

## День 3 — Агент (Agent Loop)

**Что это:** LLM + инструменты + цикл «думай → действуй → наблюдай → думай снова».
Модель сама решает, сколько шагов нужно.

**Что добавляется:**
- Планирование задачи
- Автономное выполнение шагов
- Реакция на промежуточные результаты
- Прерывание при ошибке

**Предел:** Нет памяти между разными задачами. Каждый запуск —
с нуля. Нет «личности», нет «истории».

```
Задача → [Agent: Plan → Tool → Observe → Tool → Observe → ...] → Результат
```

**Примеры:** Claude Code, OpenHands, AutoGPT, LangChain AgentExecutor

**Паттерны:**
- ReAct (Reason + Act)
- Chain-of-Thought
- Tree-of-Thought
- Self-Reflection

## День 4 — Память (Memory)

Агент без памяти — рыба без воды. Память появляется в нескольких формах:

### 4a. Кратковременная память (In-Context)
Весь разговор в контексте. Лимит: 128K–2M токенов.
Исчезает при закрытии сессии.

### 4b. Эпизодическая память (Conversation History)
Сохранение диалогов в базу. Поиск по похожим вопросам.
Технология: векторный поиск (FAISS, Chroma, Qdrant).

### 4c. Семантическая память (Knowledge Base)
Факты, документы, знания — индексированы и доступны по запросу.
Технология: RAG (Retrieval-Augmented Generation).

### 4d. Процедурная память (Skills / Tools)
Знание «как делать» — сохранённые инструкции, шаблоны, код.
Технология: SKILL.md файлы, репозитории инструментов.

### 4e. Рабочая память (State / Scratchpad)
Промежуточные результаты текущей задачи.
Технология: JSON state, Redis, файлы.

```
[Agent] ←→ [Short-term: context]
[Agent] ←→ [Episodic: vector DB]
[Agent] ←→ [Semantic: knowledge base]
[Agent] ←→ [Procedural: skills repo]
[Agent] ←→ [Working: task state]
```

## День 5 — Граф Знаний (Knowledge Graph)

**Что это:** Структурированная сеть связей между сущностями.
В отличие от плоского векторного хранилища — граф понимает
отношения, иерархии, кластеры.

**Что добавляется:**
- Сущности с типами (человек, место, концепция, документ)
- Отношения между сущностями
- Сообщества (кластеры связанных сущностей)
- Глобальный и локальный контекст

**Почему это важно:**
Плоский RAG отвечает: «найди похожее».
GraphRAG отвечает: «найди связанное, пойми контекст, синтезируй».

```
Документы → [Indexer: NER + RE] → [Knowledge Graph] → [GraphRAG Query] → Ответ
                                         ↓
                                   [Communities]
                                   [Embeddings]
                                   [Signatures]
```

**InfoM:** Q6 hex-space, 32D embeddings, TangramSignature, модулярность Q

---

### Геометрический Поток — Q6 как единый язык

Одна из редких закономерностей: в трёх независимых репозиториях
один и тот же математический объект — Q6 (6-мерный гиперкуб, 64 вершины,
изоморфный 64 гексаграммам И-Цзин) — появляется как ключевая структура:

| Репо | Применение Q6 |
|------|---------------|
| `pro2` | YiJingGPT — токены квантуются в Q6-пространстве, расстояние Хэмминга вместо косинуса |
| `infom` | Hex-адреса узлов графа, average-pool проекция N→6, HNSW поиск |
| `meta2` | TangramSignature + FractalBoundary для фрагментов документов |
| `daten22` | 16 психотипов = 4 бита = подпространство Q6 |

Это не совпадение. Q6 — геометрический язык, в котором
дискретная структура (64 состояния) + метрика (Хэмминг) + симметрия
делают его универсальным для любой категоризации.

```
64 гексаграммы И-Цзин
    = 64 вершины 6D гиперкуба
    = 64 ячейки Q6
    = адресное пространство любой сущности
```

## День 6 — Мультиагентность (Multi-Agent)

**Что это:** Несколько агентов работают параллельно или последовательно.
Каждый специализирован. Они общаются, передают задачи, проверяют друг друга.

**Паттерны:**

### Оркестратор + Воркеры
```
[Orchestrator] → [Worker: Research]
               → [Worker: Code]
               → [Worker: Review]
               ← [Results merged]
```

### Пайплайн
```
[Extractor] → [Analyzer] → [Writer] → [Reviewer] → [Output]
```

### Дебаты (Society of Mind)
```
[Agent A: proposal] ←→ [Agent B: critique] ←→ [Agent C: synthesis]
```

### Специализация
- Исследователь (web search, reading)
- Программист (code gen, execution)
- Критик (review, testing)
- Планировщик (decompose tasks)
- Коммуникатор (user interface)

**Примеры:** CrewAI, AutoGen, OpenHands, MetaGPT

---

## День 7 — Идентичность и Личность (Identity / Persona)

**Что это:** Агент приобретает устойчивые характеристики:
имя, роль, стиль, ценности, специализацию.

**Компоненты идентичности:**
- `SYSTEM_PROMPT` — роль и характер
- `MEMORY` — история опыта
- `SKILLS` — накопленные умения
- `GOALS` — долгосрочные цели
- `CONTEXT` — рабочее окружение

**Почему важно:**
Без идентичности агент — stateless функция.
С идентичностью — коллега, которому можно доверить задачу.

```yaml
# agent_identity.yaml
name: "InfoM Analyst"
role: "Knowledge graph specialist"
memory: graph_snapshots/
skills: openclaw-skills/
personality: "precise, structured, German-law aware"
context: svend4/infom
```

## День 8 — Протоколы и Стандарты (Protocols)

Когда агентов много — нужны стандарты общения.

### MCP (Model Context Protocol)
Anthropic, 2024. JSON-RPC 2.0 over stdio/HTTP.
Стандарт подключения инструментов к любой модели.
```
Model ←→ MCP Server ←→ Tools/Resources/Prompts
```

### A2A (Agent-to-Agent)
Google, 2025. Стандарт общения между агентами.
Агент A может вызвать агент B как инструмент.

### ACP (Agent Communication Protocol)
IBM/Linux Foundation, 2025. REST-based A2A протокол.

### OpenAI Agents SDK
Handoff pattern: передача управления между агентами.

```
Стандартизация = инфраструктура = экосистема
```

## День 9 — Автономность и Расписание (Autonomy / Scheduling)

**Что это:** Агент работает без запроса пользователя.
Запускается по расписанию, триггерам, событиям.

**Паттерны:**
- `cron` — каждую ночь анализировать новые документы
- `webhook` — при получении email запустить анализ
- `polling` — каждые 5 минут проверять новые данные
- `event-driven` — при коммите в репозиторий запустить агента

**Инфраструктура:**
```
[GitHub Actions] → [Agent] → [Results] → [Commit back]
[n8n / Make.com] → [Agent] → [Results] → [Webhook]
[Cron job]       → [Agent] → [Results] → [Email/Slack]
```

**Это важный переход:** Агент перестаёт быть инструментом и
становится коллегой — он работает сам.

## День 10 — Экосистема и Реестр (Ecosystem / Registry)

**Что это:** Скиллы, инструменты, агенты становятся переиспользуемыми
артефактами с версионированием, рейтингами, документацией.

**Аналогия:** npm для скиллов, GitHub для агентов.

**Примеры:**
- ClawHub (OpenClaw) — реестр SKILL.md скиллов
- Smithery — реестр MCP серверов
- LangChain Hub — реестр промптов
- HuggingFace — реестр моделей

**Структура скилла:**
```
skill-name/
  SKILL.md          — описание, команды, примеры
  tools.py          — реализация
  tests/            — тесты
  examples/         — примеры использования
  CHANGELOG.md      — история версий
```

## День 11 — Физический Мир (Embodiment / Robotics)

**Что это:** Агент получает сенсоры и актуаторы в физическом мире.

**Уровни воплощения:**
1. **Виртуальный:** браузер, файлы, API
2. **Цифровой:** IoT сенсоры, умный дом, камеры
3. **Физический:** манипуляторы, роботы, беспилотники

**Текущее состояние (2026):**
- Computer Use (Anthropic) — управление GUI
- Browser Use — автоматизация браузера
- Boston Dynamics + LLM интеграции
- Tesla Optimus, Figure 01

## День 12 — Само-улучшение (Self-Improvement)

**Что это:** Агент модифицирует свои собственные инструкции,
скиллы, код — на основе опыта.

**Формы:**
- Дообучение (Fine-tuning) на успешных взаимодействиях
- Автоматическое написание новых SKILL.md на основе решённых задач
- Рефакторинг кода агента самим агентом
- Обновление knowledge graph из новых источников

**Риски:** Alignment, безопасность, непредсказуемость.
**Статус:** Ранние эксперименты (2025–2026).

## Закон Накопления

Каждый новый элемент не заменяет предыдущий — он строится поверх.

```
Человеческое знание (День 0)
  + LLM            = Диалог с архивом
  + Tools          = Действующий архив
  + Agent Loop     = Автономный архивариус
  + Memory         = Помнящий архивариус
  + Knowledge      = Граф знаний из архива
  + Multi-agent    = Команда над архивом
  + Identity       = Специалист по архиву
  + Protocols      = Подключаемый специалист
  + Autonomy       = Самостоятельный специалист
  + Ecosystem      = Живая организация знаний
```

Мы находимся примерно на шаге **Autonomy → Ecosystem** (2025–2026).

## Место InfoM в этой карте

InfoM — это слой **Knowledge** (День 5) с элементами **Memory** (День 4),
**Protocols** (День 8) и **Autonomy** (День 9):

- `data70` → `prepare_for_infom.py` → `infom_index` → граф знаний
- GraphRAG = семантическая память + граф отношений
- Q6 hex-space = геометрическая индексация (общий язык с `pro2`, `meta2`)
- Communities = автоматическое структурирование (6 кластеров из 1105 диалогов)
- Persistence (snapshots) = долговременная память агента
- MCP (11 tools) + REST API = подключение к любому агенту/оркестратору
- GitHub Actions = автономная индексация новых документов

```
data70 (78 МБ сырых знаний)
    ↓ prepare_for_infom.py
[InfoM: NER → KnowledgeMap → Communities → Q6 → Snapshots]
    ↓ MCP / REST
[Любой агент: info7 / OpenClaw / Claude Code]
    ↓ ответ
Структурированное знание вместо сырого текста
```

InfoM решает проблему Дня 0: годы мышления, хранящиеся в текстах,
становятся живым графом — запрашиваемым, накапливаемым, передаваемым.
